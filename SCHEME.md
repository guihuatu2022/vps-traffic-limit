## 一、项目概述

### 1.1 定位

**VPS 流量监控自动锁定系统** 是一套轻量级的 Bash 脚本方案，运行在 Debian/Ubuntu 系统的 VPS 上，实现以下核心能力：

- 🎯 **自动监控**：每月定时检测 VPS 出站/入站/总流量消耗
- 🔒 **自动锁定**：流量超限时自动封锁除白名单外的所有出站流量
- 🔓 **自动解锁**：次月 1 日凌晨自动恢复网络
- 📢 **自动通知**：通过 Telegram / 企微 / 邮件 / Server酱 推送告警

### 1.2 目标用户

- 使用 **按量计费** VPS（如 GCP、AWS、Azure、Vultr、DigitalOcean 等）的个人开发者或小团队
- 需要确保 VPS **不因流量超额产生意外账单**
- 希望在流量超限时 **保留 SSH 连接**，不至于完全失联

### 1.4 设计原则

| 原则 | 说明 |
|------|------|
| **最小入侵** | 不修改系统网络配置文件，通过 iptables 规则实现控制 |
| **安全优先** | 始终保留 SSH 端口，DHCP 续租端口放行，防止失联 |
| **配置驱动** | 所有行为由配置文件控制，修改配置无需改脚本 |
| **幂等安全** | 重复运行安装脚本不会破坏已有配置 |
| **重启自愈** | 重启后规则自动恢复（靠 netfilter-persistent），lock file 清空以允许服务重新判定 |

## 三、核心概念：Lock File 机制

### 3.1 什么是 Lock File？

**Lock File** 是一个**状态标记文件**，位于 `/run/vps-traffic-limit/locked`。它不是一个真正的锁（lock/mutex），而是一个**旗帜（flag）**——用来标记"系统当前是否处于流量锁定状态"。

> 🏠 **生活比喻**：Lock file 就像你家门口的信箱上插一面小旗子。  
> - 🟢 旗子放下 = 一切正常，网络畅通  
> - 🔴 旗子立起 = 注意！流量超限，大门锁了（除 SSH 通道外）  
> - 每天早上（每次重启），旗子被自动收走（/run 清空），提醒你重新检查状态

### 3.2 为什么放在 /run/ 而不是 /var/？

| 位置 | 特性 | 为什么选它 |
|------|------|-----------|
| `/run/` | **tmpfs**（内存文件系统），重启自动清空 | 💡 **核心设计**：重启后 lock 消失 → 服务自动重新判断流量 → 若超限则重新锁，若不超限则不锁 |
| `/var/` | 持久化存储，重启后数据保留 | ❌ 重启后 lock 还在 → 即使流量正常也维持锁定，需手动处理 |
| `/tmp/` | tmpfs，但任何人都可写 | ❌ 权限过于宽松，安全性不足 |

**结论**：`/run/` 是唯一满足"重启自动清空 + 权限受控"的位置。

### 3.3 Lock 状态的 4 种矩阵

```
锁标记（lock file） vs 实际规则（iptables）的一致性矩阵：

                    ┌──────────────────────┐
                    │     iptables 规则     │
                    │  已封锁     │  未封锁  │
┌─────────┬─────────┼────────────┼──────────┤
│ Lock    │ 存在    │ ✅ 正常锁定 │ ❌ 异常  │
│ File    │         │            │（需修复）│
│         ├─────────┼────────────┼──────────┤
│         │ 不存在  │ ⚠️ 残留规则 │ ✅ 正常  │
│         │         │（需清理）  │ 正常运行 │
└─────────┴─────────┴────────────┴──────────┘
```

### 3.4 双保险校验逻辑

每次检查时，系统执行以下验证：

```
Step 1: 读取 /run/vps-traffic-limit/locked
        ├── 文件存在 → 读取锁定时的时间戳
        └── 文件不存在 → locked=false

Step 2: 检查 iptables 是否真的有封锁规则
        ├── iptables -C OUTPUT ... 检查白名单规则后的默认 DROP
        └── 实际检查规则链中的规则

Step 3: 一致性判断
        ├── lock exists + 规则存在 = ✅ 正常锁定
        ├── lock exists + 规则不存在 = 🚨 异常（lock 异常残留）
        │   └── 行为：移除 lock，重新判断流量
        ├── lock not exists + 规则存在 = ⚠️ 残留规则（重启未清理干净）
        │   └── 行为：清理规则，重新判断流量
        └── lock not exists + 规则不存在 = ✅ 正常开放
```

### 3.5 Lock 文件格式

```
# /run/vps-traffic-limit/locked
# 格式: TIMESTAMP|MODE|USED_BYTES|LIMIT_BYTES
2026-05-15 14:30:22|strict|8589934592|10737418240
```

| 字段 | 类型 | 示例 | 说明 |
|------|------|------|------|
| TIMESTAMP | 时间戳 | `2026-05-15 14:30:22` | 锁定的精确时间 |
| MODE | 字符串 | `strict` / `custom` | 锁定的模式 |
| USED_BYTES | 整数 | `8589934592` | 锁定时的已用流量（字节） |
| LIMIT_BYTES | 整数 | `10737418240` | 限额（字节） |

## 六、工程优化要点（8项）

### 优化 1：防火墙后端简化（去掉 ufw，统一 iptables）

**背景问题：**

早期方案同时支持 iptables、nftables、ufw 三种后端，导致代码大量条件分支，且 ufw 在某些环境下行为不一致（比如 ufw 默认可能覆盖已有规则）。

**改进方案：**

- 去掉 ufw 支持，**只保留 iptables**
- 检测到 iptables 缺失时自动安装 `iptables` 包
- nftables 保留为选项，但推荐 iptables（向下兼容）

**落地方式：**

```bash
# 检测 iptables
if ! command -v iptables &> /dev/null; then
    apt-get install -y iptables
fi

# 统一使用 iptables 命令（在 iptables-nft 兼容层下等效 nftables）
iptables -A OUTPUT ...  # 全平台可用
```

**关键背景**：在 Debian 12 和 Ubuntu 22.04+ 中，`iptables` 命令已经是 `iptables-nft` 兼容层，底层调用 nftables 的 API。所以 `iptables` 和 `nftables` 在底层是同一套内核接口。

---

### 优化 2：SSH 端口检测用 `sshd -T`

**背景问题：**

早期方案用 `grep Port /etc/ssh/sshd_config` 读取 SSH 端口。但这种方法有多个缺陷：

1. `sshd_config` 可能 `Include /etc/ssh/sshd_config.d/*.conf`，子配置中的 `Port` 指令会覆盖主配置
2. SSH 可能通过 systemd socket 激活（`systemctl list-sockets`），此时 sshd_config 中可能没有 Port 指令
3. `Match` 块内的配置不会体现在 `sshd -T` 的输出中——不，`sshd -T` 实际上不包含 Match 块内容

**改进方案：**

```bash
# 使用 sshd -T 获取实际生效的配置（会合并 include 所有子配置）
SSH_PORTS=$(sshd -T 2>/dev/null | grep -i '^ports ' | awk '{for(i=2;i<=NF;i++) print $i}')

# 如果为空，尝试 systemd socket 模式
if [ -z "$SSH_PORTS" ]; then
    SSH_PORTS=$(systemctl list-sockets 2>/dev/null | grep ssh | grep -oP '\d+(?=/)' | head -1)
fi

# 最终保底
[ -z "$SSH_PORTS" ] && SSH_PORTS=22
```

**落地方式**：在安装脚本的 Step 3 中调用 `sshd -T` 代替 `grep`。

---

### 优化 3：持久化方案不覆盖系统规则

**背景问题：**

早期方案通过 `iptables-save > /etc/iptables/rules.v4` 直接覆盖规则文件。这会导致一个问题：如果系统原本有其他自定义 iptables 规则（比如 Docker、fail2ban 等），覆盖操作会**删除所有已有规则**。

**改进方案：**

使用 systemd service 的排序机制：

```ini
# /etc/systemd/system/vps-traffic-limit-check.service
[Unit]
Description=VPS Traffic Limit Check
After=netfilter-persistent.service
Requires=netfilter-persistent.service

[Service]
Type=oneshot
ExecStart=/opt/vps-traffic-limit/bin/vps-traffic-limit check
ExecStop=/opt/vps-traffic-limit/bin/vps-traffic-limit restore  # 恢复而不是清空

[Install]
WantedBy=multi-user.target
```

通过 `After=netfilter-persistent.service` 确保：
- 启动时：netfilter-persistent 先加载系统规则 → 本服务再添加/检查锁定规则
- 关闭时：本服务只操作自己添加的规则（通过标记识别），不清除系统已有规则

**落地方式：**

```bash
# 添加规则时加上注释标记
iptables -A OUTPUT -p tcp --dport 22 -j ACCEPT -m comment --comment "vps-traffic-limit"

# 清除规则时只删除带有标记的行
iptables-save | grep -v "vps-traffic-limit" | iptables-restore
```

---

### 优化 4：DHCP 续租放行 67/68

**背景问题：**

如果 VPS 使用 DHCP 获取 IP 地址（很多云厂商如此），封锁出站流量会导致 DHCP 续租请求被阻止。DHCP 客户端使用 UDP 67（服务器）和 68（客户端）端口。续租失败可能导致 IP 地址丢失、SSH 断连！

**改进方案：**

无论在 strict 还是 custom 模式下，始终放行 DHCP 端口：

```bash
# iptables 规则中始终包含
iptables -A OUTPUT -p udp --sport 68 --dport 67 -j ACCEPT -m comment --comment "vps-traffic-limit DHCP lease"
```

**落地方式**：在锁定逻辑中，DHCP 端口作为**硬编码白名单**，不依赖任何配置选项，始终放行。

---

### 优化 5：Lock file 存入 /run/

**背景问题：**

见第三章节的详细分析。核心问题是重启后 Lock file 如果还在，会导致状态不一致。

**改进方案：**

- Lock file 存入 `/run/vps-traffic-limit/locked`（tmpfs）
- 重启后自动清空
- 服务启动时重新判断流量状态

**落地方式：**

```bash
# 创建运行时目录
mkdir -p /run/vps-traffic-limit

# 写入
echo "$(date '+%Y-%m-%d %H:%M:%S')|$MODE|$USED|$LIMIT" > /run/vps-traffic-limit/locked

# 读取
if [ -f /run/vps-traffic-limit/locked ]; then
    IFS='|' read -r LOCK_TIME LOCK_MODE LOCK_USED LOCK_LIMIT < /run/vps-traffic-limit/locked
fi

# 移除
rm -f /run/vps-traffic-limit/locked
```

---

### 优化 6：配置文件权限 600

**背景问题：**

配置文件包含 SMTP 密码、API Token 等敏感信息。如果权限过于宽松（如 644），系统上的其他用户可以读取这些凭证。

**改进方案：**

```bash
# 配置文件设置 600（仅 root 可读写）
install -m 600 conf/config /opt/vps-traffic-limit/conf/config

# 脚本文件设置 755
install -m 755 lib/*.sh lib/*.sh /opt/vps-traffic-limit/lib/

# 日志文件设置 640（root:adm，adm 组可读用于日志查看）
touch /opt/vps-traffic-limit/log/traffic-control.log
chmod 640 /opt/vps-traffic-limit/log/traffic-control.log
```

---

### 优化 7：日志轮转（logrotate）

**背景问题：**

日志文件会持续增长，如果 VPS 磁盘空间本就不大（如 10GB 系统盘），日志可能撑满磁盘导致系统异常。

**改进方案：**

```conf
# /etc/logrotate.d/vps-traffic-limit
/opt/vps-traffic-limit/log/traffic-control.log {
    weekly          # 每周轮转
    rotate 4        # 保留 4 周（约一个月）
    compress        # 压缩历史日志
    delaycompress   # 推迟一次压缩，方便查看最近日志
    missingok       # 日志文件不存在不报错
    notifempty      # 空文件不轮转
    create 640 root adm  # 轮转后新建文件
    postrotate
        # 通知系统释放旧文件 inode
        systemd-cat echo "vps-traffic-limit log rotated"
    endscript
}
```

---

### 优化 8：配置驱动而非代码硬编码

**背景问题：**

早期方案将接口名、端口号等参数硬编码在脚本中。每次修改都需要编辑脚本文件，容易出错且不便于管理。

**改进方案：**

所有可变参数集中在 `/opt/vps-traffic-limit/conf/config` 中，脚本只读取不修改。

```bash
# lib/config.sh
CONFIG_FILE="/opt/vps-traffic-limit/conf/config"

load_config() {
    [ -f "$CONFIG_FILE" ] || die "配置文件不存在: $CONFIG_FILE"
    source "$CONFIG_FILE"
}

# 在 /conf/config 中:
# INTERFACE="ens3"
# SSH_PORTS="22"
# LIMIT_GB=1024
# DIRECTION="egress"
# MODE="strict"
# EXTRA_PORTS=""
# NOTIFY_CHANNEL="telegram"
# NOTIFY_TOKEN="..."
# NOTIFY_CHAT_ID="..."
# FIREWALL="iptables"
```

修改配置只需编辑 `/opt/vps-traffic-limit/conf/config`，无需动脚本。修改后重载服务即可：

```bash
sudo systemctl restart vps-traffic-limit-check.timer
```

## 九、故障排查指南

### 9.1 SSH 断连了怎么办？

**现象：** 配置完成后 SSH 连接中断，无法重新连接。

**原因：** 通常是 SSH 端口配置错误，iptables 封锁规则误封了自己的 SSH 端口。

**解决方案：**

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   🔑 SSH 断连急救指南                                        │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  方法 ①：VNC / IPMI / 后台终端（推荐）               │   │
│   │  ─────────────────────────────────────────────       │   │
│   │  通过云厂商控制台的 VNC/IPMI/串行控制台登录：          │   │
│   │                                                       │   │
│   │  $ sudo iptables -F -t filter    # 清空所有 filter 规则│
│   │  $ sudo iptables -X              # 删除自定义链      │   │
│   │  $ sudo vps-traffic-limit unlock # 恢复              │   │
│   │                                                       │   │
│   │  然后检查配置中的 SSH 端口是否正确。                    │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  方法 ②：重启 VPS（网络规则会被清空）                 │   │
│   │  ─────────────────────────────────────────────       │   │
│   │  通过云厂商控制台执行"重启"操作。                        │   │
│   │  重启后 lock 文件丢失，系统会重新判断状态。              │   │
│   │  但如果磁盘上的规则持久化了，仍需进后台修复。            │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  方法 ③：联系云厂商客服（紧急）                       │   │
│   │  ─────────────────────────────────────────────       │   │
│   │  如果控制台也没有访问权限，联系技术支持。               │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │

│   · 配置前先在服务器后台打开一个备用 SSH 会话               │   │
│   · 先测试再应用：iptables 可以先用 --dry-run 模拟          │   │
└─────────────────────────────────────────────────────────────┘
```

### 9.2 误锁了怎么手动解锁？

**场景：** 流量没超限但被锁了（配置错误、测试等）。

```bash
# 方法 1：使用管理命令
sudo vps-traffic-limit unlock

# 方法 2：手动操作
sudo rm -f /run/vps-traffic-limit/locked
sudo systemctl restart vps-traffic-limit-check.timer

# 方法 3：如果管理命令不可用，直接操作 iptables
sudo iptables -F vps-traffic-limit 2>/dev/null || true
sudo iptables -X vps-traffic-limit 2>/dev/null || true
sudo iptables -F OUTPUT 2>/dev/null || true  # ⚠️ 这会清空所有 OUTPUT 规则！
```

### 9.4 日志怎么看？

```bash
# 查看实时日志
tail -f /opt/vps-traffic-limit/log/traffic-control.log

# 查看最近 50 行
tail -50 /opt/vps-traffic-limit/log/traffic-control.log

# 搜索特定关键词
grep -i "error\|warn\|fail" /opt/vps-traffic-limit/log/traffic-control.log

# 查看 systemd 服务日志
journalctl -u vps-traffic-limit-check.service --since "1 hour ago"

# 查看定时器状态
systemctl status vps-traffic-limit-check.timer
systemctl list-timers | grep vps-traffic-limit
```

### 9.5 重启后规则没恢复？

**排查流程：**

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   🔄 重启后规则未恢复排查                                     │
│                                                             │
│   重启后 iptables 规则恢复正常依赖以下链条：                   │
│                                                             │
│   ① netfilter-persistent 服务                               │
│   ────────────────────────────────────────────────           │
│   检查是否安装: dpkg -l iptables-persistent                  │
│   检查服务状态: systemctl status netfilter-persistent        │
│   检查规则文件: cat /etc/iptables/rules.v4                  │
│   如果没有: sudo netfilter-persistent save                   │
│                                                             │
│   ② 服务启动顺序是否正确                                     │
│   ────────────────────────────────────────────────           │
│   检查: systemctl list-dependencies netfilter-persistent     │
│   vps-traffic-limit-check.service 应在 netfilter 之后       │
│   修改: cat /etc/systemd/system/vps-traffic-limit-check.service│
│         [Unit] 中应有 After=netfilter-persistent.service    │
│                                                             │
│   ③ lock 文件是否清空                                        │
│   ────────────────────────────────────────────────           │
│   重启后检查: ls -la /run/vps-traffic-limit/                │
│   应该为空（/run 是 tmpfs）                                   │
│   如果存在: 说明系统未正确重启，或者 run 被另一位置覆盖         │
│                                                             │
│   ④ 手动恢复                                                │
│   ────────────────────────────────────────────────           │
│   如果重启后规则未恢复，手动执行:                              │
│   sudo systemctl restart vps-traffic-limit-check.timer      │
│   sudo /opt/vps-traffic-limit/bin/vps-traffic-limit check   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
