## 一、项目概述

### 1.1 定位

**VPS 流量监控自动锁定系统（VTL）** 是一套轻量级方案，运行在 Debian/Ubuntu 系统的 VPS 上，实现以下核心能力：

- 🎯 **自动监控**：每月定时检测 VPS 出站/入站/总流量消耗
- 🔒 **自动锁定**：流量超限时通过 **VTL-LOCK 沙盒链** 封锁非白名单出站流量，绝不触碰系统 OUTPUT 链
- 🔓 **自动解锁**：次月 1 日凌晨自动恢复网络
- 📢 **自动通知**：通过 Telegram / 企微 / 邮件 / Server酱 推送告警
- 📊 **交互菜单**：`llcx` 统一入口，新手菜单驱动 + 老手参数一步直达

### 1.2 目标用户

- 使用**按量计费** VPS（GCP、AWS、Azure、Vultr、DigitalOcean 等）的个人开发者或小团队
- 需要确保 VPS **不因流量超额产生意外账单**
- 希望在流量超限时**保留 SSH 连接**，不至于完全失联

### 1.3 核心原理

### 1.4 设计原则

| 原则 | 说明 |
|------|------|
| **沙盒隔离** | 使用独立 `VTL-LOCK` 链，绝不直接操作系统 `OUTPUT` 链 — v4.0 核心贡献 |
| **安全优先** | SSH 端口 `sport`+`dport` 双匹配，DHCP 续租端口放行，防止失联 |
| **配置驱动** | 所有行为由配置文件控制，修改配置无需改脚本 |
| **幂等安全** | 重复运行安装脚本不会破坏已有配置 |
| **重启自愈** | 重启后 lock 文件（/run tmpfs）清空 → 服务重新判定流量状态 |
| **并发安全** | `flock /run/vps-traffic-limit/vtl.lock` 防止定时器和用户命令同时操作 |
| **时区对齐** | `TZ=$PROVIDER_TIMEZONE` 按云厂商时区计费，防止跨月误判 |

## 三、核心概念：Lock File 机制

### 3.1 什么是 Lock File？

**Lock File** 是一个**状态标记文件**，位于 `/run/vps-traffic-limit/locked`。它标记"系统当前是否处于流量锁定状态"。

### 3.2 双保险机制 — 4 种状态矩阵

```
                    ┌──────────────────────┐
                    │    iptables 规则      │
                    │  已封锁     │  未封锁  │
┌─────────┬─────────┼────────────┼──────────┤
│ Lock    │ 存在    │ ✅ 正常锁定 │ ❌ 异常  │
│ File    │         │            │（需修复：│
│         │         │            │  lock 异常│
│         │         │            │  残留）  │
│         ├─────────┼────────────┼──────────┤
│         │ 不存在  │ ⚠️ 残留规则 │ ✅ 正常  │
│         │         │（需清理：  │ 正常运行 │
│         │         │  重启未清空│          │
│         │         │  或外部注入│          │
└─────────┴─────────┴────────────┴──────────┘
```

### 3.3 Lock 文件格式

```
# /run/vps-traffic-limit/locked
# 格式: TIMESTAMP|MODE|USED_BYTES|LIMIT_BYTES|DIRECTION|PROVIDER_TZ
2026-05-15 14:30:22|strict|8589934592|10737418240|egress|UTC
```

| 字段 | 类型 | 说明 |
|------|------|------|
| TIMESTAMP | 时间戳 | 锁定的精确时间（带 TZ 时区） |
| MODE | 字符串 | `strict` / `custom` |
| USED_BYTES | 整数 | 锁定时的已用流量（字节） |
| LIMIT_BYTES | 整数 | 限额（字节） |
| DIRECTION | 字符串 | 计费方向 |
| PROVIDER_TZ | 字符串 | 云厂商时区 |

### 3.4 VTL-LOCK 沙盒链设计（v5.0 核心）

```
┌─────────────────────────────────────────────────────────┐
│  VTL-LOCK 沙盒链                                        │
│                                                         │
│  锁定操作:                                               │
│  ┌─────────────────────────────────────────────────┐   │
│  │ 1. iptables -N VTL-LOCK         # 创建自定义链    │   │
│  │ 2. iptables -I OUTPUT 1 -j VTL-LOCK  # 置顶劫持  │   │
│  │ 3. iptables -A VTL-LOCK ... -j ACCEPT  # 放行规则 │   │
│  │ 4. iptables -A VTL-LOCK -j DROP  # 末尾默认拦截  │   │
│  │    # 同样操作 ip6tables（双栈）                    │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  解锁操作:                                               │
│  ┌─────────────────────────────────────────────────┐   │
│  │ 1. iptables -D OUTPUT -j VTL-LOCK  # 摘除钩子   │   │
│  │ 2. iptables -F VTL-LOCK           # 清空该链    │   │
│  │ 3. iptables -X VTL-LOCK           # 删除该链    │   │
│  │    # 同样操作 ip6tables（双栈）                    │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  优点:                                                  │
│  ✅ 绝不直接操作系统 OUTPUT 链的默认策略                 │   │
│  ✅ 清空时 +F +X 无痕拔除，系统规则零污染                │   │
│  ✅ 其他工具（Docker、fail2ban）的 OUTPUT 规则不受影响   │   │
│  ✅ 多实例安全（多个 VTL 实例各自用不同链名）            │   │
└─────────────────────────────────────────────────────────┘
```

### 3.5 锁定规则集（VTL-LOCK 链内）

#### strict 模式

```bash
# iptables -N VTL-LOCK
# iptables -I OUTPUT 1 -j VTL-LOCK

# SSH — sport + dport 双匹配！
iptables -A VTL-LOCK -p tcp --sport {SSH_PORT} -j ACCEPT -m comment --comment "vtl-ssh-return"
iptables -A VTL-LOCK -p tcp --dport {SSH_PORT} -j ACCEPT -m comment --comment "vtl-ssh-out"
# DNS
iptables -A VTL-LOCK -p udp --dport 53 -j ACCEPT -m comment --comment "vtl-dns"
iptables -A VTL-LOCK -p tcp --dport 53 -j ACCEPT -m comment --comment "vtl-dns-tcp"
# DHCP 续租
iptables -A VTL-LOCK -p udp --sport 68 --dport 67 -j ACCEPT -m comment --comment "vtl-dhcp"
# 默认拦截
iptables -A VTL-LOCK -j DROP -m comment --comment "vtl-default-drop"

# ip6tables 同理
```

#### custom 模式

```bash
# 在 strict 规则基础上，追加用户自定义端口
for port in ${EXTRA_PORTS[@]}; do
    iptables -A VTL-LOCK -p tcp --dport $port -j ACCEPT -m comment --comment "vtl-custom"
    iptables -A VTL-LOCK -p udp --dport $port -j ACCEPT -m comment --comment "vtl-custom"
done
```

## 七、工程优化要点（10 项）

### 优化 1：SSH 端口规则用 `sport` + `dport` 双匹配

**背景：** v4.0 及之前版本只用了 `--dport SSH_PORT` 放行 SSH。但 SSH 连接是双向的——服务器返回的数据包源端口是 SSH 端口，目标端口是客户端随机端口。只匹配 `dport` 在服务器回应包离开 OUTPUT 链时不会被匹配到，有可能被 DROP。

**改进：**

```bash
# ✅ 正确：双匹配
iptables -A VTL-LOCK -p tcp --sport {SSH_PORT} -j ACCEPT -m comment --comment "vtl-ssh-return"
iptables -A VTL-LOCK -p tcp --dport {SSH_PORT} -j ACCEPT -m comment --comment "vtl-ssh-out"

# ❌ 错误（v4.0）：只匹配 dport
iptables -A VTL-LOCK -p tcp --dport {SSH_PORT} -j ACCEPT  # 回应包会被 DROP！
```

### 优化 2：SSH 端口检测用 `sshd -T | grep ports`

**背景：** 直接解析 `sshd_config` 会漏掉 `Include` 子配置中的 Port 指令。

**改进：**

```bash
# ✅ 正确：sshd -T 读取实际生效的完整配置（合并所有 include）
SSH_PORTS=$(sshd -T 2>/dev/null | grep -i '^ports ' | awk '{for(i=2;i<=NF;i++) print $i}')

# ❌ 错误：只读 sshd_config（漏 Include）
# SSH_PORTS=$(grep -i '^Port' /etc/ssh/sshd_config | awk '{print $2}')
```

**回退策略：**

```bash
# 如果 sshd -T 失败（sshd 未运行），回退到 systemd socket 模式
if [ -z "$SSH_PORTS" ]; then
    SSH_PORTS=$(systemctl list-sockets 2>/dev/null | grep ssh | grep -oP '\d+(?=/)' | head -1)
fi

# 最终保底
[ -z "$SSH_PORTS" ] && SSH_PORTS=22
```

### 优化 3：持久化不覆盖系统规则

**背景：** 直接 `iptables-save > /etc/iptables/rules.v4` 会覆盖 Docker、fail2ban 等工具的规则。

**改进：**

```ini
# /etc/systemd/system/vps-traffic-limit-check.service
[Unit]
Description=VPS Traffic Limit Check
After=netfilter-persistent.service
Requires=netfilter-persistent.service
```

排序保证：
1. 系统启动 → `netfilter-persistent.service` 先加载全部系统规则（含 Docker 等）
2. 再执行 `vps-traffic-limit-check.service`（添加 VTL-LOCK 规则）

关闭时：只操作自己创建的 `VTL-LOCK` 链，不动任何系统链。

### 优化 4：通知必须先于锁定执行（时序悖论修复）

**背景：** 如果先锁定再发通知，而通知走 HTTPS（443）——但 443 已经被锁了！通知发送失败。

**改进：**

```
✅ 正确时序：
  1. 检测到超限
  2. 发送通知（此时网络还通）
  3. 执行锁定操作

❌ 错误时序（v3.0）：
  1. 检测到超限
  2. 执行锁定（通知 API 连接被阻断！）
  3. 发送通知 → 失败！
```

**落地：** core.sh 中锁定函数先调用 `send_notify "LOCKED"`，再执行 `firewall_lock`。

### 优化 5：输入注入防护

**背景：** 用户输入的端口号、命令名、接口名如果不校验，可能被用于注入攻击。

**改进：**

```bash
# 端口校验：组合校验
validate_ports() {
    local input="$1"
    # 允许字符：数字、逗号、连字符
    [[ "$input" =~ ^[0-9,\-]+$ ]] || return 1
    # 拆分校验每个端口
    IFS=',' read -ra parts <<< "$input"
    for part in "${parts[@]}"; do
        if [[ "$part" =~ ^([0-9]+)-([0-9]+)$ ]]; then
            # 端口范围
            [ "${BASH_REMATCH[1]}" -ge 1 -a "${BASH_REMATCH[1]}" -le 65535 ] || return 1
            [ "${BASH_REMATCH[2]}" -ge 1 -a "${BASH_REMATCH[2]}" -le 65535 ] || return 1
        else
            # 单个端口
            [ "$part" -ge 1 -a "$part" -le 65535 ] 2>/dev/null || return 1
        fi
    done
    return 0
}

# 接口名校验
validate_iface() {
    [[ "$1" =~ ^[a-zA-Z][a-zA-Z0-9_.-]+$ ]]
}

# 命令名校验
validate_cmdname() {
    [[ "$1" =~ ^[a-zA-Z][a-zA-Z0-9_-]{1,31}$ ]]
}
```

### 9.1 定时执行流程

```
系统启动
    │
    ├─ netfilter-persistent.service    [加载全部系统 iptables 规则]
    │       │                          [包括 Docker、fail2ban 等已有规则]

            │
            ▼
        vps-traffic-limit-check.service
            │
            ├─ flock 获取 vtl.lock     ← 并发控制
            ├─ 加载 config
            ├─ TZ=$PROVIDER_TIMEZONE date 获取厂商时区月份

            ├─ 必要时：
            │   ├─ 通知发送（先于锁定）
            │   └─ 锁定/解锁操作
            └─ flock 释放
```

### 9.2 月度重置流程

```
每月 1 日 00:00（按 PROVIDER_TIMEZONE 时区）
    │
    ▼
vps-traffic-limit-monthly.service
    │
    ├─ 检查当前锁定状态
    ├─ 如已锁定:
    │   ├─ 移除 VTL-LOCK 链
    │   ├─ 删除 Lock 文件
    │   └─ 发送解锁通知
    ├─ 发送新月份通知
    └─ 结束（交由 check 定时器持续监控）
```

### 9.3 重启自愈流程

```
重启后：
1. /run/vps-traffic-limit/ 被 tmpfs 自动清空
   → Lock file 消失 → 状态变"未锁定"
2. netfilter-persistent 加载持久化的 iptables 规则
   → 但不包含 VTL-LOCK 链（该链未持久化）
3. check 服务启动
   → 获取当前流量
   → 若超限：重新创建 VTL-LOCK 链 → 锁定 → 通知
   → 若不超限：无事发生
```

**现象：** 配置完成后 SSH 连接中断，无法重新连接。

**原因：** SSH 端口配置错误，iptables 封锁规则误封了 SSH 端口。

**急救指南：**

```
│                                                              │

│   │                                                       │   │
```

### 10.8 flock 并发冲突

**现象：** 同时运行 `llcx` 和定时器触发时，其中一个提示 `flock: 资源暂时不可用`。

**原因：** `/run/vps-traffic-limit/vtl.lock` 已被持有时另一个进程尝试获取锁，等待 30 秒后超时退出。

**解决方案：**

```bash
# 这是正常行为。锁超时表示另一个实例正在处理。
# 查看当前持有锁的进程：
lsof /run/vps-traffic-limit/vtl.lock

# 如果锁异常残留（进程已死但锁没释放），手动清除：
rm -f /run/vps-traffic-limit/vtl.lock
```
